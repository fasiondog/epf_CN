任务：评审并符合发布控制
发布控制通常由部署经理制定。这些发布控制记录了发布在投入到生产环境之前必须符合的要求。
规程：部署
用途

此任务的目的是保证对已有产品系统、产品、服务和操作没有（或最小化）负面影响。

关系
角色主执行者:
  • 开发人员
其他执行者:
输入必需:
  • 发布控制
    		可选:
    • 无
    主要描述

    发布控制描述了一个软件包在发布投入到生产中所必须遵守的最小数量的需求。这对于新的开发团队来说特别重要，因为他们可能不知道部署经理的伟大责任。事实上，部署经理负责在部署时的高层管理，保证没有不符合严格控制的事物被投入到生产中，这些控制旨在保护IT组织向内外部客户成功交付IT服务的能力。

    发布控制通常由以下事物组成：

    • 发布和部署计划
    • 回退计划
    • 发布组件定义
    • 发布包集成验证
    • 引用的配置项
    • 客户批准
    • 准备到操作和支持人员的移交
    步骤
    定位发布控制

    如果没有可用的发布控制流程程序，开发团队必须保证部署经理和/或他们的部署工程师知道在哪里找到发布控制并能够遵守它们。

    评审发布控制

    开发团队应该彻底的评审发布控制，从而理解在发布被接受投入生产环境前的预期。如果团队对控制有任何问题或疑问，团队成员应该直接和部署经理或部署工程师沟通以理解这些问题。

    保证团队发布符合这些控制

    在流程程序方面协调发布是非常正式的过程。之所以非常正式，是因为公司生产环境可能被破坏，且可能有严重的业务影响，包括：

    • 收入损失
    • 客户不满
    • 不符合法律造成的罚款
    • 失去员工生产力

    所有对发布有贡献的开发团队成员都需要遵守在流程程序层面定义的所有控制。不符合可能会有多方面的影响：

    • 整个发布被放弃，导致客户或最终用户的不满
    • Sprint/迭代开发中的多个特性没有被包含在发布中
    • 部分开发团队成员不遵守控制造成的难堪
    • 开发团队损失资金
