概念：过程的基础概念
所有过程的使用者应该熟悉这些基础的过程概念。
关系
相关元素
  • Resources for Customizing Methods
主要描述

基础元素

过程的基础元素包括

  • 工作产品（Work Product）：什么被生产
  • 任务：如何执行工作
  • 角色：谁执行工作
  • 过程：用于定义工作分解和工作流
  • 指南：模板、检查单、样例、指导书、概念，等等

这些“基础元素”是组成过程的构建模块（Building Block）。

组织元素

通过下面的元素组织上述的基础元素：

实践

实践是一个文档化的方法，用于解决一个或多个公共的问题。实践被当做采纳、使能和配置过程的“块”。实践通过上述的基础实践构建。

配置

从最终用户的角度看，一个配置就是一个待发布的选定的方法内容。

详情和样例

以下提供有关基础元素的详细信息和样例：

Work product 工作产品

工作产品有不同的形式，例如：

  • 文档，如愿景、项目计划。
  • 模型，如用例模型、设计模型。他们包含如设计类、用例、设计子系统等模型元素。
  • 数据库、工作表、以及其它形式的存储
  • 源代码和可执行文件

工作产品（Work Product）分为：工件（artifact），如果它们是具体的事物；输出产物（outcome），如果它们不是具体的；交付物（deliverable），如果它们是一个工件包。

角色

角色定义了在软件工程组织中团队中的个人或共同工作的一组人的行为和责任。注意，角色不是具体的个人；相反，角色描述了职责。个人通常可以在同一时间承担多个角色，并将在项目周期内频繁的变更角色。

样例：

  • 分析人员 - 代表客户和使用者，从利益相关人处收集并定义需求。
  • 开发人员 - 开发系统的一部分，包括设计、实现、单元测试和集成。

任务

任务是由角色执行的工作，通常被定义为一系列关于创建或更新一个或多个工作产品的步骤。

样例：

  • 开发愿景 - 为系统开发一个整体的愿景，包括捕获待解决的问题、关键利益相关人、系统的范围和边界、系统的关键特性，以及约束。
  • 策划迭代 - 定义某单个迭代的范围和责任

过程

过程将任务、工作产品和角色放在一起，并提供了结构化和时序化的信息。任务或工作产品可以被分组为高一层的活动，被称为工作分解结构（WBS，work breakdown structure）。活动或任务可以被标记为如“已计划的”来识别那些你希望分配和跟踪的工作。
